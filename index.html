§ 01

What is PySpark?

Apache Spark is an open-source, distributed computing system engineered for fast processing at large scale. PySpark is simply its Python interface.

Apache Spark
Open-source distributed computing engine for fast, large-scale data processing.
PySpark
The Python API over Spark — brings Spark's engine into Python workflows.
Parallel compute
Handles large datasets efficiently across many nodes at once.
Workload types
Batch, real-time streaming, ML, analytics, SQL querying.
Industries
Finance, healthcare, e-commerce — where scale and speed matter.
Mental model

Spark is the engine. PySpark is the steering wheel in Python. Scala, Java, and R have their own wheels attached to the same engine.

§ 02

When would we use PySpark?

Reach for PySpark when the data no longer fits — practically or economically — on a single machine.

  • AnalyticsDistributed processing with in-memory computation for speed.
  • Machine learningMLlib for scalable model training and evaluation on large datasets.
  • ETL / ELTTransforms high-volume raw data into structured, usable formats.
  • FlexibilityWorks with diverse sources — CSV, Parquet, JSON, JDBC, and more.
SQL bridge

Think of pandas as one laptop, one query and Spark as a warehouse cluster you drive with code. If your workload would OOM a laptop, it's a Spark job.

§ 04

SparkSession — the entry point

A SparkSession is the doorway into PySpark. Through it you execute queries, process data, and manage resources inside the cluster.

from pyspark.sql import SparkSession

# create or retrieve a session named "MySparkApp"
spark = (
    SparkSession.builder
        .appName("MySparkApp")
        .getOrCreate()
)
.builder
Starts configuring a new session — the method chain begins here.
.appName(name)
Labels the app so you can distinguish multiple PySpark jobs.
.getOrCreate()
Returns an existing session if one exists, else creates a fresh one.
spark
The handle you use for everything — reading data, SQL, configs.
Ex 01Create and inspect a session
# Import SparkSession from pyspark.sql
from pyspark.sql import SparkSession

# Create my_spark
my_spark = SparkSession.builder.appName("my_spark").getOrCreate()

# Print my_spark
print(my_spark)
Best practice

Always prefer SparkSession.builder.getOrCreate(). It's idempotent — safe to call anywhere in your code without spawning a second session.

SQL bridge

If Snowflake or Databricks gives you a warehouse or a cluster attachment, SparkSession is the programmatic equivalent — your open connection to the compute.

§ 05

PySpark DataFrames

A PySpark DataFrame is a distributed, table-like structure optimized for large-scale data. The surface looks like pandas; underneath, it lives across the cluster.

  • DistributedRows are partitioned across worker nodes, not sitting in local RAM.
  • API feelSimilar to pandas — same intent, different execution model.
  • InspectUse .show() to print rows to the console.
# read a CSV into a distributed DataFrame
df = spark.read.csv(data, schema=columns)

# peek at the contents
df.show()
Ex 02Load a real CSV
# Read in the CSV
census_adult = spark.read.csv("adult_reduced.csv")

# Show the DataFrame
census_adult.show()
Watch for

Pandas evaluates immediately. Spark is lazy — transformations build a plan, and nothing runs until an action like .show(), .count(), or .collect() forces it. This is the single biggest mental shift coming from SQL or pandas.

§ 06

Scalability & performance

Spark scales by distributing data and compute across a cluster. Understanding lazy evaluation, partitioning, and the Catalyst optimizer is what separates working code from fast code.

Lazy evaluation
Transformations (filter, select, withColumn) build a logical plan. Nothing executes until an action (show, count, write) triggers it.
Catalyst optimizer
Spark's query planner — rewrites your logical plan into an optimized physical plan (predicate pushdown, column pruning, join reordering).
Partitions
Data is split into chunks distributed across workers. More partitions = more parallelism, but too many = scheduling overhead.
In-memory compute
Spark caches intermediate results in RAM, avoiding repeated disk I/O — this is why it's faster than MapReduce.
# check how many partitions a DataFrame has
df.rdd.getNumPartitions()

# repartition — increase parallelism (causes a shuffle)
df_repart = df.repartition(200)

# coalesce — reduce partitions without a full shuffle
df_small = df.coalesce(10)

# cache a DataFrame in memory for repeated use
df.cache()
df.count()  # triggers the cache materialization

# explain the physical plan — see what Spark will actually do
df.filter(df.age > 30).select("name").explain(True)
Scenario — when it matters

You're running a daily ETL that reads 500 GB of Parquet, filters it, joins with a 50 MB lookup, and writes to Delta. Without .cache() on the filtered DataFrame (used in both join and an audit count), Spark reads and filters the 500 GB twice. With .cache(), the second pass hits RAM. On a 20-node cluster, that's 15 minutes saved per run.

Hard point

.repartition(n) causes a full shuffle — every row is redistributed. It's expensive. Use .coalesce(n) to reduce partitions without a shuffle. Only repartition when you're fixing severe skew or preparing for a partitioned write.

Pro tip

Run .explain(True) on your final DataFrame before production. Look for Exchange (shuffles) and BroadcastHashJoin vs SortMergeJoin — these tell you where the time goes. In Databricks, the Spark UI's SQL tab visualizes this even better.

§ 07

About DataFrames — pandas vs. Spark

Pandas and PySpark DataFrames look alike. The critical difference is where the data lives — pandas keeps it on one machine, Spark spreads it across a cluster.

Pandas
Single compute instance. Data fits in one machine's memory.
PySpark
Distributes data across many instances. Scales out, not up.
Designed for
Much larger datasets than pandas can comfortably handle.
Supports
Filtering, grouping, aggregating — plus SQL-like operations.
Watch for

The syntax looks familiar from pandas, but the execution model is different. Same verbs, different physics — Spark is lazy and distributed, pandas is eager and local.

§ 08

Creating DataFrames from files

The canonical entry point — spark.read.csv() reads a file into a DataFrame, promotes the first row to headers, and infers column types.

# load CSV with headers + type inference
df = spark.read.csv(
    "path/to/file.csv",
    header=True,
    inferSchema=True,
)
header=True
Treats the first row as column names instead of data.
inferSchema=True
Scans the file to detect types — int, double, string, etc.
Other args
sep, quote, nullValue, dateFormat, encoding, and more.
Cost note

inferSchema=True makes Spark read the file twice — once to figure out types, once to load. For large files, supplying an explicit schema= is faster and more predictable.

§ 09

Inspecting a DataFrame

Two inspection methods you'll reach for constantly — .show() to see rows, .printSchema() to see structure.

# preview the first 5 rows (default n=20; pass n to override)
df.show(5)

# display the inferred schema — column names, types, nullability
df.printSchema()
  • .show(n)Prints n rows in a formatted table. Default is 20.
  • .printSchema()Tree view of columns and their data types.
  • createDataFrame()Alternate way to build a DataFrame, e.g., from a Python list. Works, but spark.read.csv() is generally faster at scale.
Best practice

At scale, always prefer spark.read.* over createDataFrame(). The file readers are optimized for parallel I/O across the cluster.

§ 10

Basic analytics on DataFrames

Once data is loaded, aggregations summarize it — counts, sums, averages. Simple ones return a scalar; grouped ones return another DataFrame.

# simple aggregation — returns an integer
df.count()

# grouped aggregation: avg salary per gender
df.groupBy("gender").agg({"salary_usd": "avg"}).show()
.count()
Returns the total number of rows — an integer.
.groupBy(col)
Buckets rows by column values — like SQL GROUP BY.
.agg(...)
Applies one or more aggregation functions to the groups.
Typical funcs
avg, sum, min, max, stddev, count.
SQL bridge

The example is literally SELECT gender, AVG(salary_usd) FROM df GROUP BY gender. Spark's fluent API is just that query expressed as method calls.

Ex 03Load, count, group & aggregate
# Load the CSV file into a DataFrame
salaries_df = spark.read.csv("salaries.csv", header=True, inferSchema=True)

# Count the total number of rows
row_count = salaries_df.count()
print(f"Total rows: {row_count}")

# Group by company size and calculate the average of salaries
salaries_df.groupby("company_size").agg({"salary_in_usd": "avg"}).show()
salaries_df.show()
Small note

groupby and groupBy both work — they're aliases. Convention in most Spark codebases is camelCase groupBy to match the rest of the DataFrame API.

§ 11

Key operations — the SQL quartet

Four methods do most of the work — select, filter, groupBy, agg. Each maps to a SQL clause you already know.

.select(cols)
Picks specific columns. ≈ SQL SELECT.
.filter(cond)
Keeps rows matching a condition. ≈ SQL WHERE.
.groupBy(cols)
Partitions rows into groups. ≈ SQL GROUP BY.
.agg(funcs)
Aggregates each group. ≈ SUM, AVG, etc. in the SELECT.
Ex 04Filter + select
# rows where age > 50, keeping all columns
df.filter(df.age > 50).select("*").show()

# or with explicit columns
df.filter(df.age > 50).select("name", "age", "salary_usd").show()
SQL bridge

Mental translation: df.filter(df.age > 50).select("name", "age") is SELECT name, age FROM df WHERE age > 50. The pipeline order differs — Spark reads left-to-right, SQL reads top-down — but the operators are the same.

Chainable

Every method returns a new DataFrame, so you can chain: df.filter(...).groupBy(...).agg(...).show(). Nothing executes until an action such as .show(), .count(), or .collect() runs.

§ 12

Reading data formats

PySpark reads from many file formats. Each has trade-offs — pick the one that matches the shape of your data and the workload it will serve.

CSV · read.csv()
Plain-text, widely compatible. No schema enforcement — expect type inconsistencies.
JSON · read.json()
Nested or hierarchical data; high compatibility; storage-heavy at scale.
Parquet · read.parquet()
Columnar, read-optimized, schema-enforced, supports nested types. Ideal for analytics.
# three ways in, same DataFrame out
df_csv     = spark.read.csv("data.csv", header=True, inferSchema=True)
df_json    = spark.read.json("data.json")
df_parquet = spark.read.parquet("data.parquet")
Rule of thumb

For anything beyond ingestion, write intermediate and final data to Parquet — or Delta, which wraps Parquet. Columnar layout plus schema enforcement equals faster queries and fewer surprises downstream.

§ 13

Schemas & data types

Spark can infer a schema, but with complex or ambiguous data it guesses wrong. Define the schema yourself and Spark stops guessing.

  • InferenceAutomatic — Spark scans the file and picks types. Convenient, occasionally wrong.
  • ManualExplicit — you declare each column's type via StructType and StructField.
IntegerType
Whole numbers (32-bit).
LongType
Larger integers (64-bit).
FloatType
Decimal numbers (32-bit).
DoubleType
Decimal numbers (64-bit) — Spark's default for decimals.
StringType
Text values.
# import the type classes
from pyspark.sql.types import (
    StructType, StructField,
    IntegerType, StringType, DoubleType,
)

# build the schema — one StructField per column
schema = StructType([
    StructField("id",     IntegerType(), True),
    StructField("name",   StringType(),  True),
    StructField("salary", DoubleType(),  True),
])

# pass it into the reader — no more inferSchema
df = spark.read.csv("file.csv", header=True, schema=schema)
Best practice

The third argument to StructField — here True — is nullable. Leave it True unless you have a strong guarantee the column will never be null; a single stray null in a non-nullable column will fail the whole job.

Why skip inferSchema?

Two wins — speed, because Spark stops double-reading the file, and determinism, because the same file always yields the same types, avoiding silent type drift in production.

§ 14

Selection & filtering

Two fundamentals — pick the columns you want, keep only the rows you want. Both run efficiently on very large datasets.

.select(cols)
Keep specific columns. ≈ SQL SELECT.
.filter(cond)
Keep rows matching a condition. ≈ SQL WHERE.
.where(cond)
Alias of .filter() — identical behavior, choose by taste.
# pick columns
df.select("name", "salary").show()

# filter rows — .filter() and .where() are interchangeable
df.filter(df.salary > 100000).show()
df.where(df.salary > 100000).show()

# chain them together
df.filter(df.age > 30).select("name", "salary").show()
SQL bridge

If you read SQL top-down as SELECT ... FROM ... WHERE ..., Spark pipelines read left-to-right: df.filter(...).select(...). Same query, different grammar.

§ 15

Sorting & handling nulls

Two common post-load chores — put rows in order, and decide what to do with missing values.

.sort(cols)
Sort by one or more columns. ≈ SQL ORDER BY.
.orderBy(cols)
Alias of .sort() — same mechanics.
.na.drop()
Drop rows containing any null values.
.na.fill(value)
Impute — replace nulls with a value (scalar or per-column dict).
from pyspark.sql.functions import col, desc

# sort ascending by a single column
df.sort("salary").show()

# multi-column, mixed directions
df.orderBy(col("department").asc(), col("salary").desc()).show()

# drop any row with a null in any column
df.na.drop().show()

# or fill nulls instead — per-column imputation
df.na.fill({"salary": 0, "department": "unknown"}).show()
Heads up

The course frames .sort() as "simple" and .orderBy() as "complex multi-column" — in reality both are aliases and both support multiple columns. Use whichever reads better to you.

Clean vs. impute

.na.drop() is destructive — rows disappear. .na.fill() keeps rows but replaces the null with a sentinel value. Imputation is usually safer for analytics; dropping is faster and cleaner when nulls are truly unusable.

§ 16

Handling missing data — fill & drop

Real data has holes. PySpark gives you two strategies — drop the rows or fill the gaps — and lets you scope each to specific columns.

# drop rows where ANY column is null
df.na.drop().show()

# drop only if specific columns are null
df.na.drop(subset=["salary", "department"]).show()

# fill with a single value (applies to matching types)
df.na.fill(0).show()

# fill per-column with a dict
df.na.fill({"salary": 0, "department": "Unknown"}).show()
.na.drop()
Removes rows with nulls. Use subset= to target specific columns, how="all" to drop only if every column is null.
.na.fill(val)
Replaces nulls. Pass a scalar for all matching-type columns, or a dict for per-column control.
thresh=n
Optional in .na.drop() — keep the row only if it has at least n non-null values.
SQL bridge

.na.fill({"salary": 0}) is equivalent to COALESCE(salary, 0) in SQL. The .na.drop(subset=["col"]) pattern maps to WHERE col IS NOT NULL.

§ 17

Column operations — creating & renaming

DataFrames are immutable — you never edit in-place. Instead, you derive new DataFrames with added or renamed columns using withColumn and withColumnRenamed.

from pyspark.sql.functions import col, lit, upper

# add a new column (or overwrite an existing one with the same name)
df2 = df.withColumn("salary_eur", col("salary_usd") * 0.92)

# add a constant column
df3 = df.withColumn("source", lit("survey_2024"))

# rename a column
df4 = df.withColumnRenamed("salary_usd", "salary")

# transform an existing column
df5 = df.withColumn("name", upper(col("name")))
.withColumn(name, expr)
Returns a new DataFrame with an added (or replaced) column. The second argument is any Column expression.
.withColumnRenamed(old, new)
Returns a new DataFrame with the column renamed. Original DataFrame is untouched.
lit(value)
Creates a literal Column — useful for adding constants, flags, or default values to every row.
col(name)
References a column by name — used inside expressions to build derived columns.
SQL bridge

withColumn("salary_eur", col("salary_usd") * 0.92) is like SELECT *, salary_usd * 0.92 AS salary_eur FROM df. Every withColumn call is essentially adding a column to the SELECT list.

Watch for

Chaining many withColumn calls can get slow — each one creates a new projection in the plan. For adding multiple columns at once, consider using .select() with all expressions in one call.

§ 18

Advanced DataFrame operations

Beyond basic column math — casting types, conditional columns, string manipulation, and date operations are the bread and butter of real ETL work.

from pyspark.sql.functions import (
    col, when, lit, concat, substring, to_date,
    year, month, datediff, current_date, regexp_replace
)
from pyspark.sql.types import IntegerType, DoubleType

# cast a column to a different type
df = df.withColumn("age", col("age").cast(IntegerType()))

# conditional column — like SQL CASE WHEN
df = df.withColumn("tier",
    when(col("salary") >= 150000, "senior")
    .when(col("salary") >= 80000, "mid")
    .otherwise("junior")
)

# string operations
df = df.withColumn("clean_name",
    regexp_replace(col("name"), r"[^a-zA-Z\s]", "")
)

# date parsing and extraction
df = df.withColumn("hire_date", to_date(col("hire_str"), "yyyy-MM-dd"))
df = df.withColumn("tenure_days", datediff(current_date(), col("hire_date")))
df = df.withColumn("hire_year", year(col("hire_date")))
Ex 05Conditional column — flag overdue invoices
# real-world scenario: flag invoices overdue > 30 days
invoices = invoices.withColumn("status",
    when(col("paid") == True, "paid")
    .when(datediff(current_date(), col("due_date")) > 30, "overdue")
    .otherwise("pending")
)
Scenario — ETL pipeline

Raw data lands with salary as a string ("$95,000"). You need it as a double for aggregations. Chain: regexp_replace to strip $ and , → .cast(DoubleType()) → when/otherwise to bucket into bands. This three-step pattern handles 90% of "messy source column" problems.

Hard point

when().when().otherwise() evaluates top to bottom, first match wins — exactly like SQL CASE WHEN. If you forget .otherwise(), unmatched rows get null. This is a frequent source of mystery nulls in pipelines.

Pro tip

For date formats, PySpark follows Java SimpleDateFormat — "yyyy-MM-dd" not "%Y-%m-%d" (that's Python's strftime). Getting this wrong silently returns nulls instead of erroring — always .show() after a to_date() to verify.

§ 19

Unions & DataFrame combinations

Joins combine DataFrames horizontally (by key). Unions stack them vertically — appending rows from one onto another. Both are essential for consolidation pipelines.

# union — both DataFrames must have the same number of columns
# matches by POSITION, not by name
combined = df_jan.union(df_feb)

# unionByName — matches columns by NAME (safer)
combined = df_jan.unionByName(df_feb)

# unionByName with allowMissingColumns — handles schema drift
combined = df_jan.unionByName(df_feb, allowMissingColumns=True)

# deduplicate after union
combined = df_jan.unionByName(df_feb).distinct()

# drop exact-duplicate rows
df_clean = df.dropDuplicates()

# drop duplicates based on specific columns
df_clean = df.dropDuplicates(["email", "phone"])
.union(other)
Appends rows from other. Matches by column position, not name — risky if schemas differ.
.unionByName(other)
Appends rows matched by column name. Fails if a column exists in one but not the other (unless allowMissingColumns=True).
.distinct()
Removes exact duplicate rows across all columns.
.dropDuplicates(cols)
Removes duplicates based on specific columns — keeps the first occurrence.
Ex 06Monthly data consolidation
# consolidate monthly sales files into one DataFrame
from functools import reduce

months = ["jan", "feb", "mar"]
dfs = [spark.read.parquet(f"sales/{m}.parquet") for m in months]

# stack them all
all_sales = reduce(lambda a, b: a.unionByName(b), dfs)
print(f"Total rows: {all_sales.count()}")
Hard point

.union() matches by position — if df_jan has (name, salary) and df_feb has (salary, name), the values silently end up in the wrong columns. Always use .unionByName() in production.

Scenario — schema drift

Upstream adds a new discount_pct column in March's file but January and February don't have it. With .union() this crashes. With .unionByName(allowMissingColumns=True), the missing column fills with nulls — which you can then .na.fill(0). This is the safe pattern for data lakes where schemas evolve over time.

§ 20

DataFrame joins

Combining DataFrames by a shared key — the same concept as SQL joins, with the same types and the same gotchas.

# inner join — only matching rows survive
joined = flights.join(airports, flights.dest == airports.faa, "inner")

# left outer — keep all from left, null-pad from right
joined = flights.join(airports, flights.dest == airports.faa, "left")

# if the key column has the same name in both DataFrames
joined = df1.join(df2, "id")  # auto-matches on "id", drops duplicate col
.join(other, on, how)
Joins two DataFrames. on is the condition or column name, how is the join type.
"inner"
Only rows with matches in both sides. This is the default.
"left" / "left_outer"
All rows from the left DataFrame, nulls where no match on the right.
"right" / "full"
Right outer and full outer — same semantics as SQL.
Ex 07Join flights with airports
# join flights to airports on destination code
flights_with_dest = flights_df.join(
    airports_df,
    flights_df.dest == airports_df.faa,
    "left"
)
flights_with_dest.show(5)
Watch for

When both sides have a column with the same name (e.g., both have "id"), Spark keeps both — causing ambiguous references downstream. Either pass a string (.join(df2, "id")) which auto-deduplicates, or .drop() the duplicate after the join.

SQL bridge

flights.join(airports, flights.dest == airports.faa, "left") maps directly to SELECT * FROM flights LEFT JOIN airports ON flights.dest = airports.faa.

Scenario — multi-table enrichment

You have a transactions DataFrame (100M rows), a customers lookup (500K rows), and a products lookup (10K rows). Chain: transactions.join(broadcast(products), "product_id").join(broadcast(customers), "customer_id"). Broadcast both small tables to avoid two expensive shuffles. This pattern covers 80% of star-schema enrichment jobs.

Hard point

A left join with a mismatched key type (e.g., string "123" vs integer 123) silently produces zero matches — every right-side column becomes null. Always verify key types with .printSchema() on both sides before joining. Cast if needed: df.withColumn("id", col("id").cast("int")).

Pro tip

After a join, immediately .select() only the columns you need. Carrying unused columns through the rest of the pipeline wastes memory and shuffle bandwidth — Spark doesn't auto-prune them after a join.

§ 21

User-Defined Functions (UDFs)

When built-in functions aren't enough, you write your own. A UDF wraps a Python function so Spark can apply it row-by-row across the cluster.

from pyspark.sql.functions import udf
from pyspark.sql.types import IntegerType, StringType

# define a regular Python function
def age_bucket(age):
    if age is None:
        return "unknown"
    return "senior" if age >= 60 else "adult" if age >= 18 else "minor"

# register it as a UDF — must declare the return type
age_bucket_udf = udf(age_bucket, StringType())

# apply it
df.withColumn("bucket", age_bucket_udf(df.age)).show()
udf(fn, returnType)
Wraps a Python function for distributed execution. You must specify the return type.
Return type
Always declare it — StringType(), IntegerType(), DoubleType(), etc. Spark cannot infer it.
Null handling
UDFs receive None for nulls — guard against it or you'll get runtime errors.
Ex 08Integer UDF
# UDF that returns an integer
def double_salary(salary):
    return int(salary * 2) if salary is not None else None

double_udf = udf(double_salary, IntegerType())
df.withColumn("doubled", double_udf(df.salary)).show()
Performance

Standard UDFs serialize data between the JVM and Python row by row — this is slow. Use built-in functions whenever possible. When you must use custom logic, prefer Pandas UDFs (next section) for vectorized performance.

Scenario — when you actually need a UDF

You need to validate IDs against a custom checksum algorithm your company uses (e.g., Luhn check on account numbers). No built-in function exists. Write the validation as a UDF returning BooleanType(), apply it, then filter: df.filter(validate_udf(col("account_id"))). This is one of the few cases where a standard UDF is justified.

Pro tip

Before writing a UDF, check pyspark.sql.functions — there are 200+ built-in functions covering math, string, date, regex, array, map, and JSON operations. Nine times out of ten, what you need already exists and runs 10–50× faster than a UDF.

§ 22

Pandas UDFs — vectorized performance

Pandas UDFs (a.k.a. vectorized UDFs) operate on batches of rows as pandas Series instead of one-at-a-time. The payoff is dramatically faster execution.

import pandas as pd
from pyspark.sql.functions import pandas_udf
from pyspark.sql.types import DoubleType

# decorator style — operates on a pandas Series, returns a pandas Series
@pandas_udf(DoubleType())
def salary_to_eur(salary: pd.Series) -> pd.Series:
    return salary * 0.92

# apply like any column function
df.withColumn("salary_eur", salary_to_eur(df.salary_usd)).show()
@pandas_udf(type)
Decorator that marks a function as vectorized. Input and output are pandas Series.
Arrow-backed
Uses Apache Arrow for zero-copy transfer between JVM and Python — much faster than row-by-row serialization.
Batch operation
Spark sends chunks of rows. The function processes them with pandas/NumPy vectorized ops.
UDF vs Pandas UDF

Regular UDF: one row at a time → Python overhead per row → slow at scale. Pandas UDF: batch of rows → vectorized NumPy/pandas ops → 10–100× faster. Always prefer Pandas UDFs when your logic can be expressed as Series operations.

Type hint

The type hints (pd.Series -> pd.Series) aren't just documentation — PySpark uses them to determine the UDF variant. Omitting them can cause confusing runtime errors.

§ 23

Resilient Distributed Datasets (RDDs)

RDDs are Spark's lowest-level abstraction — an immutable, distributed collection of objects. DataFrames are built on top of them. You rarely use RDDs directly, but understanding them matters.

# create an RDD from a Python list
rdd = spark.sparkContext.parallelize([1, 2, 3, 4, 5])

# create from a text file
rdd_text = spark.sparkContext.textFile("data.txt")

# collect — pulls all data back to the driver
rdd.collect()   # → [1, 2, 3, 4, 5]

# take — returns only the first n elements
rdd.take(3)      # → [1, 2, 3]
RDD
Resilient Distributed Dataset — Spark's foundational data structure. Immutable, partitioned across nodes.
.parallelize(list)
Creates an RDD from a local Python collection, distributing it across the cluster.
.textFile(path)
Reads a text file into an RDD where each line is one element.
.collect()
Action — returns the entire RDD to the driver as a Python list. Careful with large datasets — can OOM.
.take(n)
Safe alternative — returns just the first n elements without pulling everything.
RDD vs DataFrame

DataFrames know about columns and types — Spark can optimize their query plans (Catalyst optimizer). RDDs are unstructured — Spark treats them as opaque objects and can't optimize. For analytics, always prefer DataFrames.

When to use RDDs

Low-level transformations that don't fit the DataFrame model, processing unstructured text data, or when you need fine-grained control over partitioning. Otherwise, stick with DataFrames.

Scenario — when RDDs still matter

You're ingesting raw log files where each line is a semi-structured string that doesn't parse cleanly into columns. Use sc.textFile() → .map() with a custom parser → .filter() malformed lines → convert to DataFrame once the data is clean. RDDs are the right tool for the "messy text → structured table" bridge.

Hard point

.collect() pulls everything to the driver's memory. On a 50 GB RDD, this will crash your driver node. Use .take(n) for inspection and .foreach() or .saveAsTextFile() for output. In interviews, mentioning this shows you understand the driver vs. executor distinction.

§ 24

Spark SQL — query DataFrames with SQL

Register a DataFrame as a temporary view, then query it with plain SQL. The best of both worlds — Python for orchestration, SQL for analytics.

# register a DataFrame as a temp view
df.createOrReplaceTempView("employees")

# now query it with SQL
result = spark.sql("""
    SELECT department, AVG(salary) AS avg_salary
    FROM employees
    WHERE age > 30
    GROUP BY department
    ORDER BY avg_salary DESC
""")
result.show()
.createOrReplaceTempView(name)
Registers the DataFrame as a named SQL table scoped to this SparkSession. Idempotent — safe to call repeatedly.
spark.sql(query)
Executes a SQL string and returns the result as a DataFrame. Supports full Spark SQL syntax.
Temp view scope
Exists only within the current SparkSession. Use createGlobalTempView to share across sessions.
Ex 09Query on a temp view
# register
flights_df.createOrReplaceTempView("flights")

# query — all familiar SQL
long_flights = spark.sql("""
    SELECT origin, dest, air_time
    FROM flights
    WHERE air_time > 300
    ORDER BY air_time DESC
""")
long_flights.show(10)
Best of both

This is the hybrid pattern — use Python/PySpark for loading, cleaning, and orchestration; switch to SQL for complex analytical queries where SQL syntax is more natural. Both produce the same optimized execution plan under the hood.

Scenario — migration from data warehouse

Your team has 200 existing SQL queries in Snowflake. Instead of rewriting each in the DataFrame API, register your DataFrames as temp views and run the SQL as-is via spark.sql(). Migrate the orchestration to Python (scheduling, error handling, alerts) while keeping the SQL logic identical. This cuts migration time by 80%.

Pro tip

Use f-strings to parameterize SQL queries: spark.sql(f"SELECT * FROM orders WHERE order_date = '{run_date}'"). For production, prefer spark.sql() with .where() chained after to inject dynamic filters safely — avoids SQL injection patterns in shared notebooks.

§ 25

Analytics with SQL on DataFrames

Once you can write SQL against DataFrames, the full toolkit opens up — window functions, subqueries, CTEs, and complex multi-table analytics.

# analytics — multiple aggregations in one SQL query
spark.sql("""
    SELECT 
        department,
        COUNT(*) AS headcount,
        ROUND(AVG(salary), 2) AS avg_salary,
        MAX(salary) AS max_salary,
        MIN(salary) AS min_salary
    FROM employees
    GROUP BY department
    HAVING COUNT(*) > 5
    ORDER BY avg_salary DESC
""").show()

# window function example — rank within department
spark.sql("""
    SELECT name, department, salary,
           RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rank
    FROM employees
""").show()
SQL bridge

This is literally the same SQL you'd write in Snowflake or Databricks SQL. The only difference is you're running it via spark.sql() instead of a SQL worksheet. Same optimizer, same semantics.

Mixing approaches

The DataFrame returned by spark.sql() is a normal PySpark DataFrame — you can chain .filter(), .withColumn(), or .write on it. SQL and the DataFrame API are fully interoperable.

§ 26

PySpark aggregations — deep dive

Beyond the basic .agg() — multiple aggregation functions, named outputs, and complex grouped computations.

from pyspark.sql.functions import (
    col, count, avg, sum, min, max, round as spark_round
)

# multiple aggregations in one call
df.groupBy("department").agg(
    count("*").alias("headcount"),
    spark_round(avg("salary"), 2).alias("avg_salary"),
    max("salary").alias("top_salary"),
    min("salary").alias("low_salary"),
).orderBy("avg_salary", ascending=False).show()

# no groupBy — aggregate the entire DataFrame
df.agg(
    count("*").alias("total_rows"),
    avg("salary").alias("global_avg"),
).show()
.alias(name)
Renames the aggregation output column — like AS in SQL.
Function imports
Use pyspark.sql.functions for count, avg, sum, min, max, stddev, collect_list, etc.
Dict vs function style
.agg({"col": "avg"}) is quick but limited. Function style (.agg(avg("col").alias("name"))) is more flexible.
Pro tip

Watch the import name collision — Python's built-in round, sum, min, max conflict with PySpark's. Either import with an alias (from pyspark.sql.functions import round as spark_round) or use F.round() via import pyspark.sql.functions as F.

§ 27

Complex aggregations — pivot, rollup, cube

Beyond groupBy().agg() — pivot turns row values into columns, rollup adds subtotals, and cube gives you every possible grouping combination.

from pyspark.sql.functions import sum, avg, count, collect_list

# PIVOT — turn row values into columns
# like a spreadsheet crosstab
pivot_df = df.groupBy("department") \
    .pivot("quarter", ["Q1", "Q2", "Q3", "Q4"]) \
    .agg(sum("revenue"))
pivot_df.show()
# → department | Q1    | Q2    | Q3    | Q4
# → Sales      | 50000 | 62000 | 58000 | 71000

# ROLLUP — subtotals at each group level + grand total
df.rollup("department", "role") \
    .agg(sum("salary").alias("total_salary")) \
    .orderBy("department", "role").show()
# → rows with nulls = subtotal/grand-total rows

# CUBE — all possible grouping combinations
df.cube("department", "location") \
    .agg(count("*").alias("headcount")) \
    .show()

# collect_list — aggregate into an array
df.groupBy("department").agg(
    collect_list("name").alias("members")
).show(truncate=False)
.pivot(col, values)
Rotates distinct values of a column into separate columns. Always pass the values list explicitly for performance.
.rollup(cols)
Like groupBy but adds hierarchical subtotals. Nulls in group columns indicate subtotal rows.
.cube(cols)
Like rollup but generates every combination of grouping columns — the full multi-dimensional summary.
collect_list(col)
Collects all values per group into an array. Use collect_set for unique values only.
Ex 10Revenue pivot table
# build a pivot: revenue by region × product category
report = sales_df \
    .groupBy("region") \
    .pivot("category", ["Electronics", "Clothing", "Food"]) \
    .agg(sum("revenue").alias("total"))

report.na.fill(0).show()
Scenario — executive reporting

Finance asks for a summary: total salary spend by department, with subtotals per department and a grand total. .rollup("department", "role").agg(sum("salary")) gives you all three levels in one query — no manual UNION ALL with a totals row.

Hard point

Pivot without specifying the values list forces Spark to do a pre-scan of all distinct values — an extra full pass over the data. Always pass .pivot("col", ["A", "B", "C"]) explicitly. In Databricks notebooks, df.select("col").distinct().show() first, then hardcode the list.

Pro tip

Use collect_list + explode as a pair — collect_list groups into arrays, explode flattens arrays back into rows. Together they enable sessionization, basket analysis, and time-window aggregations that are awkward with pure SQL.

§ 28

Aggregations in RDDs

RDDs aggregate differently — using functional primitives like map, reduce, reduceByKey, and countByKey instead of SQL-style group-by.

# setup — RDD of (key, value) pairs
rdd = spark.sparkContext.parallelize([
    ("engineering", 95000),
    ("marketing",   72000),
    ("engineering", 110000),
    ("marketing",   68000),
    ("engineering", 88000),
])

# reduce — single aggregate over all values
total = rdd.map(lambda x: x[1]).reduce(lambda a, b: a + b)
# → 433000

# reduceByKey — aggregate per key (like GROUP BY + SUM)
totals_by_dept = rdd.reduceByKey(lambda a, b: a + b)
totals_by_dept.collect()
# → [('engineering', 293000), ('marketing', 140000)]

# countByKey — count per key
rdd.countByKey()
# → {'engineering': 3, 'marketing': 2}
.map(fn)
Transformation — applies a function to each element, returns a new RDD.
.reduce(fn)
Action — combines all elements using a binary function. Returns a single value to the driver.
.reduceByKey(fn)
Groups by key and applies the reduce function within each group. Stays distributed.
.countByKey()
Action — counts occurrences per key. Returns a Python dict to the driver.
SQL bridge

rdd.reduceByKey(lambda a, b: a + b) is the RDD equivalent of SELECT dept, SUM(salary) FROM t GROUP BY dept. Same result, more manual wiring — which is why DataFrames are preferred.

§ 29

Broadcast variables & optimization

When a small dataset is used by every worker — like a lookup table — broadcasting it sends one copy to each node instead of shuffling it with every task.

from pyspark.sql.functions import broadcast

# broadcast the smaller DataFrame in a join
result = large_df.join(
    broadcast(small_lookup_df),
    "key_column"
)
result.show()

# RDD-level broadcast variable
lookup = {"US": "United States", "IN": "India", "DE": "Germany"}
bc_lookup = spark.sparkContext.broadcast(lookup)

# use inside an RDD transformation
rdd.map(lambda row: (row, bc_lookup.value.get(row, "Unknown"))).collect()
broadcast(df)
Hints Spark to send the small DataFrame to all nodes before the join — avoids expensive shuffles.
sc.broadcast(obj)
RDD-level — broadcasts a Python object (dict, list) to every worker. Access via .value.
When to use
One side of a join is small (< ~100 MB). Lookup tables, config maps, dimension tables.
Avoid when
Both sides are large — broadcasting a big table wastes memory on every node.
Performance impact

Without broadcast: Spark shuffles both datasets across the network to co-locate matching keys — expensive. With broadcast: the small table is sent once, and the large table stays in place. For a 10 GB ⋈ 50 MB join, this can be the difference between minutes and seconds.

SQL bridge

In Databricks SQL, this maps to the /*+ BROADCAST(small_table) */ hint. Spark often auto-broadcasts tables under 10 MB (configurable via spark.sql.autoBroadcastJoinThreshold), but being explicit is safer for important jobs.

Hard point

Broadcasting a table that's too large (> 500 MB) will cause the driver to OOM — it has to collect and serialize the entire table before sending it to workers. If you're not sure, check: spark.sql("SELECT COUNT(*), SUM(size) FROM table").show() or look at the DataFrame size in the Spark UI's Storage tab.

Pro tip

For RDD-level broadcast variables, remember to call bc_var.unpersist() when you're done — broadcast variables stay in memory on every executor until explicitly cleared. In long-running streaming jobs or notebooks with many cells, this prevents memory leaks.

§ 30

Bringing it all together — end-to-end pipeline

A realistic pipeline that ties together everything: read raw data, clean it, join with a lookup, aggregate, and write the result — using best practices from every section.

Ex 11Full pipeline — flight delay analysis
from pyspark.sql import SparkSession
from pyspark.sql.functions import (
    col, avg, count, when, broadcast, round as spark_round
)
from pyspark.sql.types import (
    StructType, StructField, StringType, IntegerType, DoubleType
)

# 1 — SESSION
spark = SparkSession.builder.appName("FlightDelays").getOrCreate()

# 2 — READ with explicit schema (skip inferSchema for speed)
flight_schema = StructType([
    StructField("flight_id",   IntegerType(), True),
    StructField("origin",      StringType(),  True),
    StructField("dest",        StringType(),  True),
    StructField("delay_min",   DoubleType(),  True),
    StructField("air_time",    DoubleType(),  True),
])
flights = spark.read.csv("flights.csv", header=True, schema=flight_schema)

airports = spark.read.parquet("airports.parquet")  # small lookup

# 3 — CLEAN: drop nulls in critical columns, fill others
flights_clean = (
    flights
    .na.drop(subset=["origin", "dest"])
    .na.fill({"delay_min": 0.0})
    .withColumn("is_delayed",
        when(col("delay_min") > 15, True).otherwise(False)
    )
)

# 4 — JOIN with broadcast (airports is small)
enriched = flights_clean.join(
    broadcast(airports),
    flights_clean.dest == airports.faa,
    "left"
)

# 5 — AGGREGATE: delay stats per destination city
report = (
    enriched
    .groupBy("city", "state")
    .agg(
        count("*").alias("total_flights"),
        spark_round(avg("delay_min"), 1).alias("avg_delay"),
        count(when(col("is_delayed"), True)).alias("delayed_count"),
    )
    .withColumn("delay_pct",
        spark_round(col("delayed_count") / col("total_flights") * 100, 1)
    )
    .orderBy(col("avg_delay").desc())
)

# 6 — WRITE to Delta (or Parquet)
report.write.mode("overwrite").parquet("output/delay_report")

# quick preview
report.show(10)
  • § 04SparkSession — entry point to the cluster.
  • § 13Explicit schema — faster, deterministic reads.
  • § 16na.drop + na.fill — clean missing data.
  • § 18when/otherwise — conditional columns.
  • § 29broadcast — optimize the small-table join.
  • § 26groupBy + agg with named outputs.
Pro tip — production checklist

Before promoting any pipeline: (1) .explain(True) to verify broadcast join was used, (2) check partition count on the output, (3) add a .count() assertion on the input and output for data quality, (4) parameterize paths and dates so the same notebook runs for any day.

Scenario — Databricks production

In a Databricks job, replace .write.parquet() with .write.format("delta").mode("merge").saveAsTable("gold.delay_report") to get ACID transactions, time travel, and schema enforcement for free. The pipeline logic stays identical — only the sink changes.
